## Current Design Implications

- The most realistic first network model is not "server renders the whole Unity game headless." It is a server-authoritative run/world ledger: run seed, dungeon graph, room activation, pickups, equipment assignment, damage/death, and persistent cult/world deltas.
- A command-prompt server can probably own JSON-like state and deterministic decisions, but actual Unity scene activation, physics, AI, animation, and player controllers should stay in clients for now.
- P2 emulation should likely lean on vanilla co-op where possible. The hard part is not spawning a second local player; it is ensuring the remote client, host, and server agree on room graph, rewards, pickups, damage, and late-join equipment.
- The likely join trick is role mapping plus host-side co-op emulation: the first server client remains the host Lamb, the second server client is treated as the owner of the host game's P2 body, and that P2 body is eventually driven by remote client telemetry/commands once mutation is safe.
- Late-join P2 equipment should be treated as a server-owned assignment rule: if P2 has no persisted equipment for this run/session, copy the current authoritative P1 weapon and curse; if P2 already has persisted equipment, restore that instead of re-copying P1.
- Save rerouting should be staged, not immediate. The safer target is server-owned `server_worlds/<worldId>/world.json` plus a future save mirror/snapshot channel. Directly pointing live game saves at a remote/network path risks corruption. The plugin should first dry-run the intended redirect path, then later intercept/pull/push saves at controlled load/save boundaries.
- Reserve COTL's visible UI slot 5 as the current online/server-world convention. In the save files this is raw slot `4` (`slot_4.*` / `meta_4.*`), so the server should default to `serverSaveSlot=4` and warn with `saveSlotOk=False` instead of mutating saves when a client is not on that slot.

## Server Ledger Prototype

- The first useful validation target is replaying `cotlonline-trace-20260714-124348.jsonl` and confirming the ledger reports the same P2 late-join copy rule seen manually in the trace.

- `0.4.1` adds a periodic `live.heartbeat` event so a UDP listener started after the game can still learn plugin version, scene, run, coop state, and player count.
- After the `13:16` UDP run, the ledger now derives command-shaped server decisions from the event stream: `register_reward`, `claim_reward`, and `assign_equipment`. This is still observe-only, but it proves the command-prompt side can maintain a compact authoritative equipment state that matches the live game.
- `0.4.2` adds a separate `sync.player_motion` stream for p0/p1 coordinate testing. This is intentionally separate from low-frequency `sync.player_state`: movement can be sampled at 5Hz or 10Hz without making every runtime snapshot more expensive.
- `0.4.3` adds `clientId` and `sessionId` to trace events and changes the server ledger to key players as `client:p0`, `client:p1`, etc. This is required before LAN testing, because each joining machine has its own local Lamb/player zero.

The roster also assigns provisional roles: first active client is `host-lamb`, second active client is `remote-p2`, and additional clients are `pending`. The server replies to player motion with `server.remote_players`; the client records it as `bridge.remote_players` and shows the latest remote player position/state in the F8 overlay. ## Phase 5 Direction

- This route is not Parsec, Steam Remote Play, or screen sharing. Each client keeps running its own local game. The server owns session/world/save/seed authority, while the host DLL owns the real vanilla co-op P2 body that remote input can eventually drive.
- The reason to prefer a host-side vanilla P2 body first is that COTL already knows how to route co-op combat, hitboxes, damage, co-op rewards, revive behavior, and P2 dungeon-specific assets through that body. If this works, we avoid manually rebuilding every co-op rule in the server.
- The current build reserves the host P2 body and can feed remote input intent into P2's vanilla `InputManager.Gameplay` axis/button path. The next layer is not more coordinate sync; it is world/session authority: seed enforcement, transition gating, and save mirroring so P1 and P2 cannot enter different logical spaces.
- Follower interactions are currently unsafe for meaningful sync. Followers roam and choose interactions independently on each local instance, so a remote P2 interaction with a host-visible follower can diverge even when player movement is close. The likely next authority step is to block remote P2 follower/interactable actions during movement tests, then re-enable specific interactions through host/server-owned commands.
- Faith, devotion, follower actions, resource deposits, building changes, and similar cult/world changes should be captured as world-event commands. The server should own ordering and replay/validation for these events; the connector DLL should apply the accepted command locally. This is different from movement smoothing and should not be attempted as per-frame follower position sync.
- Copying the same save into the reserved slot is necessary but not sufficient. Once both Unity simulations wake the cult scene, follower AI, thoughts, faith, tasks, and structure inventories can diverge locally. `0.5.11` separates raw save-slot matching from active cult-state matching so the next step can target command replay or remote simulation suppression instead of more movement smoothing.
- Dungeon reward/loadout sync is a separate but nearer-term requirement than full follower sync. The server already observes podium generation, pickup claims, `PlayerWeapon.SetWeapon`, and spell assignment. The next implementation should send a small authoritative loadout packet back to both clients so P1/P2 item readiness matches before vanilla dungeon doors evaluate co-op completion.
- Seeing both players on both clients should be staged. The host already uses a real vanilla P2 body for the remote player. The remote-p2 client should first use a non-authoritative host-P1 visual marker or ghost. Spawning a real local co-op body on the remote client is a later test because it can trigger local co-op camera, item, follower, and save behavior.
- Phase7 starts that later test in an explicitly non-authoritative way. The remote client spawns a local vanilla P2 body only when it is assigned `remote-p2`, blocks that mirror from local transitions, and treats the body as a host-lamb visual mirror rather than a gameplay owner. If it destabilizes camera or local co-op systems, disable `RemoteHostVisualMirror` and keep using markers until camera/focus patches are split out.
- `PlayerFarming.Update()` removes non-lamb P2 when its Rewired player has no joystick or keyboard. `0.5.0` temporarily sets `CoopManager.Instance.temporaryDisableRemoval` only while an active host reservation needs it, and logs that as `p2Hold=True`. `0.5.10` applies the same idea to the remote-p2 Phase7 mirror and blocks late vanilla removal calls directly if the hold is lost.
- Dungeon seed authority remains a separate required step. The server roster now includes the observed `runSeed`, but it does not enforce it yet. Seed enforcement needs to happen before `BiomeGenerator.GenerateRoutine()` uses `BiomeGenerator.Seed` to create the room graph.

## Phase 4 Direction

- Keep the next build server-side unless a specific enforcement test is needed. The immediate target is an authority ledger that records run seed, world graph, rewards, reward claims, equipment, and damage/death as command-shaped state changes.
- Treat `claim_reward` as the source of ownership. `coopPodium` is useful metadata for understanding how the room generated, but it cannot decide ownership.
- Treat `observed_game_assignment` as confirmation. When the server later sends commands back to the plugin, the game-side assignment should be compared against the server decision and repaired only if it diverges.
- The first plugin receiver should be narrow: read a command from localhost and log the intended repair without mutating equipment. Only after that dry-run matches multiple traces should it actually call weapon/curse assignment paths.
- Movement should stay client-mod plus server, not server-only. The plugin sees local Unity transforms/controller state and can smooth or repair movement; the command-prompt server should own ordering, routing, authority, and validation.
- LAN testing shape: host PC runs the server ledger, each game install runs the diagnostics/connector plugin, and non-host clients set `UdpHost` to the host PC's LAN IP. The first LAN goal is only proving distinct `client:p0` streams arrive without overwriting each other.
- The next bridge success criterion is that both real clients have local trace lines for `bridge.roster` that include both client ids. That will show each game instance is receiving server state, not only sending telemetry.
- After `0.4.5`, the faster success criterion is visual: each game should show the same two-client roster in the overlay. Local traces still remain the audit proof.
- After `0.4.6`, the next validation is world matching: host and laptop should show the same `worldId` in the overlay. If both are using copied/matching saves, both should show `match=True`; if not, the laptop should show `match=False` and the server `world.json` should preserve both fingerprints for comparison.
- After `0.4.7`, the clean test target is stricter: both machines should be on UI save slot 5 / raw file slot 4, both should show `slotOk=True`, both should show the same `worldId`, and matching copied saves should show `match=True`.
- After `0.4.9`, both machines should also show `remote relay` in the F8 overlay. Moving or dodging on one machine should update the other machine's remote player line with that client's position, state, hp, sequence, and age.
- After `0.5.2`, host testing should focus on whether the reserved goat/P2 enters native `Moving` and dodge/attack states when the laptop moves, dodges, or attacks. If the host P2 moves with animations but still lags, the next bottleneck is packet cadence/intent transport rather than P2 spawning.
- After `0.5.4`, host testing should verify three things: `input-spawn-snap` appears once after F9, laptop input drives P2 without immediate offset, and walking into building/door triggers with the remote P2 logs a transition block instead of moving the host to a different room.

## Phase 3 Targets

- Trace `DataManager.SetNewRun` and `DataManager.AddNewDungeonSeed` to capture the run seed lifecycle.
- Trace `BiomeGenerator` room graph mutation points: random walk, entrance/exit, lock/key, special rooms, custom rooms, and prefab instantiation.
- Trace `Interaction_WeaponPickUp`, `PlayerWeapon.SetWeapon`, and `PlayerSpells.SetSpell` with short caller stacks so P2 mid-dungeon gear assignment has an attributable source.
- Keep Phase 3 read-only. No combat hooks and no state mutations.
